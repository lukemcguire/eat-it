# Phase 04.1: Recipe Ingredient Management - Research

**Researched:** 2026-03-04
**Domain:** FastAPI/SQLModel CRUD endpoints for nested resources
**Confidence:** HIGH

## Summary

This phase adds CRUD endpoints for ingredient groups and ingredients within recipes.
The models (`Ingredient`, `IngredientGroup`) already exist in the codebase with
foreign key relationships but have no API exposure. The implementation follows
established patterns from the existing recipes router, using nested routes under
`/recipes/{recipe_id}/` with bulk operations support for full ingredient replacement.

**Primary recommendation:** Follow the existing recipe router patterns exactly,
using the same session management, error handling, and schema structures.
Add `display_order` field via Alembic migration.

## User Constraints (from CONTEXT.md)

### Implementation Decisions

#### URL Structure

- **Nested under recipes:** All endpoints are under `/recipes/{recipe_id}/`
- **Groups:** `/recipes/{recipe_id}/groups` - CRUD for ingredient groups
- **Ingredients:** `/recipes/{recipe_id}/ingredients` - direct access, group_id in payload
  Rationale: Users don't navigate to ingredients via groups in URLs - they edit
  ingredients within a recipe context.

#### Default Group Behavior

- **Auto-created:** A default "Ingredients" group is created when the first
  ingredient is added to a recipe with no existing groups.
- **Auto-deleted:** Empty groups are removed when all their ingredients are deleted.
- **Always grouped:** Every ingredient belongs to a group. No "ungrouped"
  ingredients exist.
- **Default name:** The auto-created group is named "Ingredients".

#### Cascade on Delete

- **Group -> Ingredients:** Deleting a group cascade-deletes all its ingredients.
- **Recipe -> Groups:** Deleting a recipe cascade-deletes all groups (existing FK
  behavior).
- **Empty groups:** When all ingredients in a group are deleted, the group itself
  is deleted automatically.

#### Bulk Operations

- **Single endpoint:** `PUT /recipes/{recipe_id}/ingredients` replaces all groups
  and ingredients.
- **Full replacement:** Payload contains complete state of all groups with nested
  ingredients.
- **ID handling:** Existing items include their `id`, new items have `id: null`.
- **Deletion:** Groups/ingredients not in payload are deleted.
- **Order preserved:** `display_order` field on ingredients determines order
  within each group.

#### Ordering

- **display_order field:** Integer field on Ingredient model for manual ordering.
- **Within group only:** Ordering is relative to other ingredients in the group.
- **Gaps allowed:** No renumbering on delete - gaps in display_order are fine.

#### Response Behavior

- **Return item only:** Create/update endpoints return just the created/updated
  item (not full recipe).
- **Bulk returns full state:** The bulk PUT endpoint returns the complete
  groups+ingredients structure.

### Claude's Discretion

- Exact validation error messages
- Whether to validate group names as unique within a recipe
- Default display_order value (suggest: creation order index)
- Transaction boundary for bulk operations (all-or-nothing)

### Pattern 1: Nested Resource Router

**What:** Router mounted under recipes prefix for nested resources.

**When to use:** All ingredient/group endpoints are recipe-scoped.

**Example:**

```python
# src/eat_it/routers/ingredients.py
from fastapi import APIRouter, Depends, HTTPException, status
from sqlmodel import Session, select

from eat_it.database import get_session
from eat_it.models.recipe import Ingredient, IngredientGroup, Recipe
from eat_it.schemas.ingredient import (
    IngredientCreate,
    IngredientPublic,
    IngredientUpdate,
    IngredientGroupCreate,
    IngredientGroupPublic,
    IngredientGroupUpdate,
)

router = APIRouter(tags=["ingredients"])

@router.post(
    "/groups",
    response_model=IngredientGroupPublic,
    status_code=status.HTTP_201_CREATED,
)
def create_group(
    recipe_id: int,
    group: IngredientGroupCreate,
    session: Session = Depends(get_session),
) -> IngredientGroup:
    """Create a new ingredient group for a recipe."""
    # Verify recipe exists
    recipe = session.get(Recipe, recipe_id)
    if not recipe:
        raise HTTPException(status_code=404, detail="Recipe not found")

    db_group = IngredientGroup(
        recipe_id=recipe_id,
        name=group.name,
    )
    session.add(db_group)
    session.commit()
    session.refresh(db_group)
    return db_group
```

### Pattern 2: Bulk Replace with Transaction

**What:** Single PUT endpoint that replaces all groups and ingredients atomically.

**When to use:** Full ingredient list replacement for edit-all-at-once UX.

**Example:**

```python
# src/eat_it/routers/ingredients.py
@router.put(
    "/",
    response_model=IngredientsBulkResponse,
)
def replace_all_ingredients(
    recipe_id: int,
    payload: IngredientsBulkRequest,
    session: Session = Depends(get_session),
) -> dict:
    """Replace all groups and ingredients for a recipe."""
    # Verify recipe exists
    recipe = session.get(Recipe, recipe_id)
    if not recipe:
        raise HTTPException(status_code=404, detail="Recipe not found")

    # Get existing data
    existing_groups = session.exec(
        select(IngredientGroup).where(IngredientGroup.recipe_id == recipe_id)
    ).all()
    existing_group_ids = {g.id for g in existing_groups}
    existing_ingredient_ids = set()
    for g in existing_groups:
        ingredients = session.exec(
            select(Ingredient).where(Ingredient.group_id == g.id)
        ).all()
        existing_ingredient_ids.update(i.id for i in ingredients)

    # Track IDs from payload
    payload_group_ids = {g.id for g in payload.groups if g.id is not None}
    payload_ingredient_ids = set()
    for g in payload.groups:
        payload_ingredient_ids.update(
            i.id for i in g.ingredients if i.id is not None
        )

    # Delete groups not in payload (cascade deletes ingredients)
    for group in existing_groups:
        if group.id not in payload_group_ids:
            session.delete(group)

    # Create/update groups and ingredients
    for group_data in payload.groups:
        if group_data.id and group_data.id in existing_group_ids:
            # Update existing group
            group = session.get(IngredientGroup, group_data.id)
            group.name = group_data.name
            session.add(group)
        else:
            # Create new group
            group = IngredientGroup(
                recipe_id=recipe_id,
                name=group_data.name,
            )
            session.add(group)
            session.flush()  # Get ID for ingredient FK

        # Handle ingredients for this group
        # ... similar pattern for ingredients

    session.commit()

    # Return full state
    return build_bulk_response(session, recipe_id)
```

### Anti-Patterns to Avoid

- **Don't use async endpoints:** All existing CRUD endpoints are synchronous
  (FastAPI handles thread pool automatically). Keep consistency.
- **Don't load embedding model:** Ingredient endpoints don't need semantic
  search. Skip embedding-related code entirely.
- **Don't return full recipe:** Per CONTEXT.md, single-item operations return
  just the item, not the whole recipe.
- **Don't renumber display_order:** Gaps are allowed. Don't compact on delete.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Request validation | Custom validators | Pydantic schemas with Field constraints | Proven pattern in existing schemas |
| 404 handling | Custom error middleware | HTTPException with status code | Consistent with existing routers |
| Session management | Per-request engine | Depends(get_session) | Established pattern, thread-safe |
| Bulk operation rollback | Manual try/catch | Session transaction (session.rollback on exception) | SQLModel handles automatically |

**Key insight:** The existing codebase has battle-tested patterns. Copy them
directly rather than introducing new approaches.

## Common Pitfalls

### Pitfall 1: Forgetting Recipe Existence Check

**What goes wrong:** Creating groups/ingredients for non-existent recipe_id
causes foreign key constraint violation or orphaned data.

**Why it happens:** Developer assumes recipe_id is valid.

**How to avoid:** Every endpoint must verify `session.get(Recipe, recipe_id)`
before any group/ingredient operations.

**Warning signs:** 500 errors with foreign key violation messages.

### Pitfall 2: Not Flushing for FK Assignment

**What goes wrong:** Creating group and ingredients in same transaction fails
because group.id is None when creating ingredients.

**Why it happens:** SQLModel doesn't auto-flush to get IDs.

**How to avoid:** Call `session.flush()` after adding new group, before creating
ingredients that need its ID.

**Warning signs:** "NOT NULL constraint failed: ingredients.group_id"

### Pitfall 3: Missing Cascade Delete Configuration

**What goes wrong:** Deleting a group leaves orphaned ingredients, causing
constraint errors or data corruption.

**Why it happens:** SQLite foreign keys don't cascade by default.

**How to avoid:** The existing migration has proper FK setup. For new
relationships, ensure migrations include `ON DELETE CASCADE` or handle
deletion in application code.

**Implementation note:** CONTEXT.md specifies cascade in application logic
for group->ingredients, not database-level cascade. Delete ingredients
explicitly before deleting group.

### Pitfall 4: Display Order Default Value

**What goes wrong:** Creating ingredients without display_order leaves them
unordered or causes insertion errors.

**Why it happens:** Field has no default in model.

**How to avoid:** Set display_order to len(existing_ingredients) in creation
endpoint, or use server_default in migration.

**Recommendation:** Calculate in endpoint code for flexibility:
`display_order=len(current_group_ingredients)`

### Pitfall 5: Empty Group Cleanup Race

**What goes wrong:** After deleting all ingredients from a group, the empty
group remains if cleanup logic is in wrong place.

**Why it happens:** Cleanup runs before transaction commits, or is forgotten.

**How to avoid:** Check for empty groups at end of delete operations, delete
them before commit. For bulk operations, handle this after all deletions.

### Schema Pattern (from schemas/recipe.py)

```python
from pydantic import BaseModel, ConfigDict, Field
from datetime import datetime
from typing import Optional

class IngredientBase(BaseModel):
    """Base fields for ingredient schemas."""
    quantity: Optional[float] = None
    unit: Optional[str] = None
    name: str = Field(..., min_length=1, max_length=500)
    preparation: Optional[str] = None
    raw: str = Field(..., min_length=1)
    display_order: int = Field(0, ge=0)

class IngredientCreate(IngredientBase):
    """Schema for creating ingredient."""
    group_id: int  # Required - every ingredient belongs to a group

class IngredientUpdate(BaseModel):
    """Schema for partial ingredient update."""
    quantity: Optional[float] = None
    unit: Optional[str] = None
    name: Optional[str] = Field(None, min_length=1, max_length=500)
    preparation: Optional[str] = None
    raw: Optional[str] = Field(None, min_length=1)
    display_order: Optional[int] = Field(None, ge=0)
    group_id: Optional[int] = None  # Allow moving to different group

class IngredientPublic(IngredientBase):
    """Schema for ingredient response."""
    model_config = ConfigDict(from_attributes=True)

    id: int
    group_id: int
    created_at: datetime
    updated_at: datetime
```

## Open Questions

1. **Group name uniqueness within recipe?**
   - What we know: CONTEXT.md leaves this to Claude's discretion.
   - Recommendation: Do NOT enforce uniqueness. Two groups named "Sauce" is
     valid user intent. Frontend can warn if desired.

2. **Default display_order calculation?**
   - What we know: CONTEXT.md suggests creation order index.
   - Recommendation: Set to `len(existing_ingredients_in_group)` on creation.
     Simple, predictable, no gaps until deletion (which is fine per CONTEXT.md).

3. **Transaction boundary for bulk PUT?**
   - What we know: CONTEXT.md leaves this to Claude's discretion.
   - Recommendation: All-or-nothing. Wrap in try/except with session.rollback()
     on exception. This matches user expectation for "save all changes" operation.
