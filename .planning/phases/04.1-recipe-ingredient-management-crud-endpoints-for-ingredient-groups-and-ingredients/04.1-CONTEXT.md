# Phase 04.1: Recipe Ingredient Management - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

CRUD endpoints for managing ingredient groups and individual ingredients within recipes.
The models already exist (IngredientGroup, Ingredient) but have no API endpoints.
This phase exposes them through RESTful endpoints with bulk operations support.

</domain>

<decisions>
## Implementation Decisions

### URL Structure

- **Nested under recipes:** All endpoints are under `/recipes/{recipe_id}/`
- **Groups:** `/recipes/{recipe_id}/groups` - CRUD for ingredient groups
- **Ingredients:** `/recipes/{recipe_id}/ingredients` - direct access, group_id in payload
  Rationale: Users don't navigate to ingredients via groups in URLs - they edit ingredients within a recipe context.

### Default Group Behavior

- **Auto-created:** A default "Ingredients" group is created when the first ingredient is added to a recipe with no existing groups.
- **Auto-deleted:** Empty groups are removed when all their ingredients are deleted.
- **Always grouped:** Every ingredient belongs to a group. No "ungrouped" ingredients exist.
- **Default name:** The auto-created group is named "Ingredients".

### Cascade on Delete

- **Group → Ingredients:** Deleting a group cascade-deletes all its ingredients.
- **Recipe → Groups:** Deleting a recipe cascade-deletes all groups (existing FK behavior).
- **Empty groups:** When all ingredients in a group are deleted, the group itself is deleted automatically.

### Bulk Operations

- **Single endpoint:** `PUT /recipes/{recipe_id}/ingredients` replaces all groups and ingredients.
- **Full replacement:** Payload contains complete state of all groups with nested ingredients.
- **ID handling:** Existing items include their `id`, new items have `id: null`.
- **Deletion:** Groups/ingredients not in payload are deleted.
- **Order preserved:** `display_order` field on ingredients determines order within each group.

### Ordering

- **display_order field:** Integer field on Ingredient model for manual ordering.
- **Within group only:** Ordering is relative to other ingredients in the group.
- **Gaps allowed:** No renumbering on delete - gaps in display_order are fine.

### Response Behavior

- **Return item only:** Create/update endpoints return just the created/updated item (not full recipe).
- **Bulk returns full state:** The bulk PUT endpoint returns the complete groups+ingredients structure.

### Claude's Discretion

- Exact validation error messages
- Whether to validate group names as unique within a recipe
- Default display_order value (suggest: creation order index)
- Transaction boundary for bulk operations (all-or-nothing)

</decisions>

<specifics>
## Specific Ideas

- Frontend UX: Click "edit" → all fields become editable inline → drag to reorder → hit "update" to save
- Each group has a "+" button to add ingredients
- Each ingredient has a drag handle for reordering
- Bottom of page has "+" button to add new section/group
- User shouldn't need to think about groups - they can just add ingredients and the default group handles it

</specifics>
