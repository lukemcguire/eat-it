# Phase 05.1: Ingredient Management Frontend - Research

**Researched:** 2026-03-05
**Domain:** React frontend with @dnd-kit drag-and-drop, TanStack Query
**Confidence:** HIGH

## Summary

This phase implements the frontend interface for managing ingredient groups and
individual ingredients within recipes. The backend Phase 04.1 already provides all
necessary CRUD endpoints plus a bulk replace endpoint. The frontend will use
@dnd-kit for drag-and-drop reordering with a desktop-first editing experience.

**Primary recommendation:** Use @dnd-kit with SortableContext for nested group and
ingredient reordering. Implement an explicit edit mode with a single bulk save
operation using the existing `PUT /recipes/{recipe_id}/ingredients` endpoint.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Edit Mode Pattern:**
- Entry: Explicit "Edit" button in the header/ingredients section
- Save: Single "Save Changes" button triggers bulk PUT to
  `/recipes/{recipe_id}/ingredients`
- Cancel: Warn on unsaved changes when user cancels or navigates away
  ("Discard unsaved changes?" dialog)
- Visual indicator: Edit button swaps to Cancel + Save buttons; header
  shows "Editing Ingredients" or similar
- Scope: Desktop-first editing for MVP; mobile shows view-only mode

**Group Visibility:**
- Single group: Display as flat list with no group header
- Multiple groups: Show group names as section headers with ingredients
  nested beneath
- Full group CRUD: Create, rename, delete groups via inline actions
- "Add Group" button: At bottom of ingredients section
- Delete confirmation: "Delete group and its N ingredients?" before
  cascade-deleting
- Group reorder: Drag groups to reorder (desktop)
- Move ingredients between groups: Drag ingredient to different group
  section (desktop)
- Empty groups: Keep until explicitly deleted

**Reordering UX:**
- Drag handle: Grip icon on left of each row
- Library: @dnd-kit for drag-and-drop implementation
- Scope: Desktop only (mobile editing deferred)

**Mobile Layout (View Mode):**
- Ingredient display: One row per ingredient, format "1 C. flour"
  (quantity+unit, then name)
- Groups: Section headers, same pattern as desktop but stacked vertically
- No editing: Edit button hidden on mobile; view-only for MVP

### Claude's Discretion

- Exact styling/spacing for ingredient rows and group headers
- Drop indicator animation during drag
- Accessibility (keyboard navigation for drag-drop)
- Error handling for save failures
- Loading states during save

### Deferred Ideas (OUT OF SCOPE)

- Mobile editing: Touch-friendly ingredient editing on small screens
  (drag-drop alternatives, inline editing on mobile)
- Keyboard shortcuts: Quick navigation between ingredients, bulk actions
- Ingredient autocomplete: Suggest common ingredients while typing

### Pattern 2: @dnd-kit Sortable with DragOverlay

**What:** Use useSortable hook with separate presentational component for DragOverlay
**When to use:** Implementing drag-and-drop reordering

```typescript
// Source: https://docs.dndkit.com/presets/sortable
import { useSortable } from '@dnd-kit/sortable';
import { CSS } from '@dnd-kit/utilities';

function SortableIngredientRow({ id, ingredient }: Props) {
  const {
    attributes,
    listeners,
    setNodeRef,
    transform,
    transition,
    isDragging,
  } = useSortable({ id });

  const style = {
    transform: CSS.Transform.toString(transform),
    transition,
    opacity: isDragging ? 0.5 : 1,
  };

  return (
    <div ref={setNodeRef} style={style} {...attributes}>
      <DragHandle {...listeners} />
      <IngredientRow ingredient={ingredient} />
    </div>
  );
}
```

### Pattern 3: Multiple Containers (Groups) with Cross-Container Drag

**What:** Multiple SortableContext providers under single DndContext
**When to use:** Dragging ingredients between groups

```typescript
// Source: https://docs.dndkit.com/presets/sortable
function IngredientSection() {
  return (
    <DndContext
      sensors={sensors}
      collisionDetection={closestCenter}
      onDragEnd={handleDragEnd}
      onDragOver={handleDragOver}
    >
      {groups.map((group) => (
        <SortableContext
          key={group.id}
          items={group.ingredients.map(i => i.id)}
          strategy={verticalListSortingStrategy}
        >
          <IngredientGroup group={group} />
        </SortableContext>
      ))}
    </DndContext>
  );
}
```

### Anti-Patterns to Avoid

- **Rendering useSortable component in DragOverlay:** Causes id collision.
  Use a separate presentational component instead.
- **Not using closestCenter collision detection:** Default rectangle
  intersection is too strict for sortable lists.
- **Updating items prop out of render order:** SortableContext items must
  match render order or drag behavior breaks.
- **Storing display_order in frontend state during drag:** Use array index
  for local reordering, compute display_order only on save.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Drag-and-drop from scratch | Custom mouse/touch handlers | @dnd-kit | Accessibility, edge cases, animations |
| Optimistic updates | Manual state sync | TanStack Query's built-in support | Race conditions, rollback |
| Unsaved changes dialog | Custom beforeunload | react-router-dom useBlocker | Browser compatibility |

**Key insight:** @dnd-kit handles all the edge cases (touch, keyboard,
accessibility, collision detection) that are deceptively complex to implement
correctly.

## Common Pitfalls

### Pitfall 1: DragOverlay Component Collision

**What goes wrong:** Rendering the same sortable component in DragOverlay causes
unexpected behavior with duplicate ids.

**Why it happens:** useSortable internally uses useDraggable; two components
with same id calling useDraggable causes collision.

**How to avoid:** Create a separate presentational component (without useSortable)
for the DragOverlay. Use forwardRef pattern.

```typescript
// CORRECT: Separate presentational component
const IngredientRowOverlay = forwardRef(({ ingredient }, ref) => (
  <div ref={ref} className="...">
    {/* Same visual as IngredientRow but no useSortable */}
  </div>
));
```

**Warning signs:** Dragged item disappears, console errors about duplicate ids.

### Pitfall 2: SortableContext Items Out of Order

**What goes wrong:** Drag behavior is erratic, items jump to wrong positions.

**Why it happens:** The items array passed to SortableContext must be in the
same order as rendered items.

**How to avoid:** Always derive items from the same state used for rendering:

```typescript
// CORRECT: items matches render order
const ingredientIds = group.ingredients.map(i => i.id);
return (
  <SortableContext items={ingredientIds} strategy={verticalListSortingStrategy}>
    {group.ingredients.map((ingredient) => (
      <SortableIngredientRow key={ingredient.id} ingredient={ingredient} />
    ))}
  </SortableContext>
);
```

**Warning signs:** Items snap to unexpected positions during drag.

### Pitfall 3: Backend Empty Group Auto-Delete vs Frontend Empty Group Display

**What goes wrong:** User creates empty group, saves, and group disappears.

**Why it happens:** CONTEXT.md notes backend Phase 04.1 auto-deletes empty
groups. Frontend wants to keep empty groups until explicitly deleted.

**How to avoid:** Two options:
1. **Frontend workaround:** Filter out empty groups before sending to bulk
   endpoint, show warning to user.
2. **Backend adjustment:** Modify Phase 04.1 to not auto-delete during bulk
   operations (only on individual ingredient delete).

**Recommendation:** Use frontend workaround for MVP. Show "Empty groups will
not be saved" helper text.

**Warning signs:** Groups with no ingredients disappear after save.

### Pitfall 4: Missing Keyboard Accessibility

**What goes wrong:** Drag-and-drop only works with mouse, not keyboard.

**Why it happens:** Forgetting to configure KeyboardSensor with sortableKeyboardCoordinates.

**How to avoid:** Always include keyboard sensor:

```typescript
const sensors = useSensors(
  useSensor(PointerSensor),
  useSensor(KeyboardSensor, {
    coordinateGetter: sortableKeyboardCoordinates,
  })
);
```

**Warning signs:** Cannot navigate drag with Tab/Arrow keys.

### TanStack Query Hook (frontend/src/hooks/useIngredients.ts)

```typescript
// Source: Pattern from useRecipes.ts
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { apiClient } from '@/lib/api';
import type {
  IngredientGroupWithIngredients,
  IngredientsBulkRequest,
  IngredientsBulkResponse,
} from '@/types/ingredient';

export function useIngredientGroups(recipeId: number) {
  return useQuery({
    queryKey: ['recipes', recipeId, 'ingredient-groups'],
    queryFn: () =>
      apiClient<IngredientGroupWithIngredients[]>(
        `/recipes/${recipeId}/groups`
      ),
    enabled: !!recipeId,
  });
}

export function useBulkReplaceIngredients(recipeId: number) {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: (payload: IngredientsBulkRequest) =>
      apiClient<IngredientsBulkResponse>(
        `/recipes/${recipeId}/ingredients`,
        {
          method: 'PUT',
          body: JSON.stringify(payload),
        }
      ),
    onSuccess: () => {
      queryClient.invalidateQueries({
        queryKey: ['recipes', recipeId, 'ingredient-groups'],
      });
      queryClient.invalidateQueries({ queryKey: ['recipes', recipeId] });
    },
  });
}
```

### Single vs Multiple Group Display

```typescript
// Source: CONTEXT.md specifications
function IngredientSection({ groups }: { groups: IngredientGroupWithIngredients[] }) {
  const isSingleGroup = groups.length <= 1;

  if (isSingleGroup) {
    // Flat list with no group header
    const ingredients = groups[0]?.ingredients ?? [];
    return (
      <div>
        {ingredients.map((ing) => (
          <IngredientRow key={ing.id} ingredient={ing} />
        ))}
      </div>
    );
  }

  // Multiple groups with headers
  return (
    <div>
      {groups.map((group) => (
        <div key={group.id}>
          <h3 className="font-bold text-lg">{group.name}</h3>
          {group.ingredients.map((ing) => (
            <IngredientRow key={ing.id} ingredient={ing} />
          ))}
        </div>
      ))}
    </div>
  );
}
```

## Open Questions

1. **Empty Group Persistence**
   - What we know: Backend auto-deletes empty groups (ING-05).
   - What's unclear: Should frontend prevent empty group creation or just
     warn?
   - Recommendation: Show "Empty groups will not be saved" helper text when
     group has no ingredients. Filter empty groups before bulk save.

2. **Cross-Group Drag Timing**
   - What we know: onDragOver can detect when ingredient moves between groups.
   - What's unclear: Should update happen during drag or only on drop?
   - Recommendation: Update local state on drop (onDragEnd) for simplicity.
     Cross-container visual feedback during drag is nice-to-have.
